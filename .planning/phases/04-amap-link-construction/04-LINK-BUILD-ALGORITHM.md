# Phase 4 链接构造算法（L0 / L1..Ln）

## 目标

将确认顺序节点转换为稳定可验证的高德链接输出，满足 `from/to` 显式、`callnative=1`、URL 编码与兼容兜底要求。

## 输入与输出

- 输入：按用户确认顺序的节点列表（示例：`A1 -> A3 -> A2 -> B3 -> B2 -> B1`）。
- 输出：
  - `L0`: 总段链接（首节点到末节点）。
  - `L1..Ln`: 相邻节点分段链接，顺序与输入一致。

## Step 1: 前置校验与停止条件

- 检查终点是否存在；缺终点立即停止（D-11）。
- 检查起点来源：优先用户显式起点，其次当前位置（D-09）。
- 若无起点且无法定位，立即停止（D-10）。
- 若校验失败，仅返回缺失说明，不生成无效链接。

## Step 2: 生成 L0 总段链接

- `L0` 使用确认序列首节点为 `from`，末节点为 `to`（D-08）。
- 主路径固定 `https://uri.amap.com/navigation`（D-01）。
- 参数写入 `callnative=1` 且值执行 URL 编码（D-02）。

## Step 3: 生成 L1..Ln 分段链接

- 按确认顺序遍历相邻节点对：
  - `L1 = A1 -> A3`
  - `L2 = A3 -> A2`
  - `L3 = A2 -> B3`
  - `L4 = B3 -> B2`
  - `L5 = B2 -> B1`
- 禁止自动重排（D-05, D-06）。
- 每条分段链接必须显式 `from/to`（D-07）。

## Step 4: 参数填充与编码

- 每条链接都包含 `callnative=1`（D-02）。
- 每个参数值进行 URL 编码，避免中文与保留字符破坏 URI。
- 坐标存在时按 GCJ-02 写入定位值；文本仅作展示补充（D-04）。

## Step 5: scheme 条件输出与 https 兜底

- 默认只输出 https（D-12）。
- 用户明确要求 scheme 时，输出 scheme + https 成对结果。
- 无论是否有 scheme，https 都必须保留（D-13）。

## Worked Example

输入顺序：`A1 -> A3 -> A2 -> B3 -> B2 -> B1`

- `L0`: `A1 -> B1`
- `L1`: `A1 -> A3`
- `L2`: `A3 -> A2`
- `L3`: `A2 -> B3`
- `L4`: `B3 -> B2`
- `L5`: `B2 -> B1`

该命名与顺序用于执行与验收统一比对。

## 需求映射

| 需求 ID | 算法落点 |
|---|---|
| AMAP-01 | Step 2/3 产出完整可点击 `https://uri.amap.com/...` |
| AMAP-02 | Step 4 强制 `callnative=1` + URL 编码 |
| AMAP-03 | Step 4 坐标优先（GCJ-02） |
| AMAP-04 | Step 5 scheme 可选且 https 兜底 |
