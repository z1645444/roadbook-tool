# Phase 2 POI Strategy

## 1. 候选生成规则（D-01 / D-02 / D-03 / D-04）

- **D-01（POI-01）**：默认输出动态 `3-5` 个候选 POI，保持结果可比较、可排序、可复用。
- **D-02（POI-01）**：当输入关键信息不足时，降级输出 `2-3` 个候选，并在条目与摘要中显式标注`信息不足`，不阻断流程。
- **D-03（POI-03）**：候选补位优先使用“同城同类”高相关项，保证语义连续性和城市一致性。
- **D-04（POI-03）**：候选不足时不放宽硬约束，只允许放宽软偏好（如次要偏好权重），避免错误扩展结果空间。

执行要点：
- 硬约束示例：城市硬限制、用户明确排除条件、必须包含的出行目的。
- 软偏好示例：热门度倾向、步行距离偏好、商圈偏好。

## 2. 排序优先级（D-05 / D-06 / D-07 / D-08）

- **D-05（POI-03）**：先过`同城优先`门控，再按`语义匹配 > 偏好匹配 > 可达性 > 热门度`排序。
- **D-06（POI-02）**：同分第一 tie-breaker 使用理由可解释性，优先保留能被单句理由稳定说明的候选。
- **D-07（POI-03）**：热门度仅作兜底信号，不得压过语义匹配与偏好匹配。
- **D-08（POI-03）**：跨城候选默认后置，仅在同城池不足或语义更强时进入尾部候选。

排序约束：
- 同城候选整体排在跨城候选之前。
- 在跨城场景中仍保持可解释顺序，避免热门度主导。

## 3. 推荐理由规则（D-09 / D-10 / D-11 / D-12）

- **D-09（POI-02）**：每个候选必须提供`单句`推荐理由，避免多句扩展造成风格漂移。
- **D-10（POI-02）**：理由固定模板为`匹配点 + 场景价值`。
- **D-11（POI-02）**：默认不强制输出“为什么没选其他候选”，保持核心说明简洁稳定。
- **D-12（POI-02）**：仅在低置信场景显示置信标记，常规场景不展示置信标签。

模板一致性要求：
- `匹配点`用于描述用户意图与候选之间的直接对应。
- `场景价值`用于描述该候选在当前路线目标中的实际收益。

## 4. 多样性与去重（D-13 / D-14 / D-15 / D-16）

- **D-13（POI-03）**：`Top3` 候选至少覆盖 2 个类别，避免结果扎堆单一类型。
- **D-14（POI-03）**：重复判定定义为`同名 + 近地址`，作为文档级统一去重标准。
- **D-15（POI-01）**：去重后用同城高分候选补位，保持候选数量与排序稳定。
- **D-16（POI-03）**：同类候选连续最多 2 个，超过时触发重排或补位。

去重与补位流程：
- 先做同名 + 近地址去重，再做类别连续性检查。
- 若去重导致数量不足，优先回填同城高相关候选，再考虑跨城尾部回填。

## 5. 低置信兜底（D-17 / D-18 / D-19 / D-20）

- **D-17（POI-01）**：缺关键字段或存在语义冲突时触发`低置信`模式。
- **D-18（POI-02）**：低置信模式下仍输出候选，每个候选显式标注低置信状态。
- **D-19（POI-01）**：低置信场景追加 `1-2` 条`补充信息建议`，只给最关键字段建议。
- **D-20（POI-03）**：允许下游继续处理，但必须附`风险提示`，标明候选排序可信度下降。

低置信输出要求：
- 降级时仍满足 2-3 个候选，且每项可解释。
- 补充信息建议优先字段：精确目的地、时间窗、交通偏好、预算边界。
